📢 Live In-App Banner

Writes to notifications/latest. All users on the site see the banner immediately via Firestore onSnapshot.

📊 Live Fetch Progress

Controls the progress bar on simulation.html via notifications/fetch_progress. Update percent, counts, status (running / paused / completed), and timing fields manually while data is being fetched.

📅 Induction Schedule

Publishes portal step windows to the Schedule tab on simulation.html via notifications/induction_schedule in Firestore. Do not fetch the portal API from this public site — requests from GitHub Pages are blocked and would expose the private endpoint. Fetch the schedule locally (browser on the portal while logged in, Postman, etc.), paste the JSON Table array or full response below, then save.

✅ Profile Verification Status

Publishes per-applicant verification status to the Candidates tab on simulation.html via notifications/profile_status in Firestore. Supports the v2 grouped format (statusTypes + entries) for multiple status types (e.g. 131 = Verification Round 01, 132 = Amendment Process). Paste full JSON or an entries array — same workflow as schedule: fetch locally, paste JSON, save.

Status types metadata (click to edit)

JSON object mapping type ID → {label, statusLabels}. Leave empty to fall back to single-type save.

📌 Community Chat — Pinned Announcement

Shows a pinned banner at the top of community chat on simulation.html via notifications/chat_pin. Only one announcement at a time — use for rules, deadlines, or important updates.

📝 Simulation Notifications Feed

Edits the simulation page notification feed using the same schema as data/notifications.json. On GitHub Pages this is stored in notifications/simulation_feed in Firestore, because the admin portal cannot directly rewrite static repo files from the browser.

📱 Web Push Subscribers

Users who clicked "Alerts" → allowed browser notifications. Their FCM tokens are stored in push_subscriptions.

✎ Candidate Revisions

Tracked Fields

💼 Job Board Sync

Crawl the public listing pages, parse each job post, and merge only new or changed documents into the jobs Firestore collection. Each job carries a stable fingerprint — when content already matches an existing doc the sync skips that write. The public Jobs tab (on the main app) renders from this collection in real time so visitors see the latest synced board instantly.
